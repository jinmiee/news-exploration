# 성능: 분석 결과 DB 캐싱 효과

## 배경

뉴스 차트/상세/감정/연관어 페이지의 분석은 대부분 **형태소 전처리(Okt) + KoBERT 임베딩 유사도** 계산 비용이다.
이 프로젝트는 분석을 사용자 요청마다 수행하지 않고, **스케줄러(APScheduler)가 사전 계산해 결과를 MongoDB에 저장**하고,
요청 시에는 결과를 **조회만** 한다(Base64 차트 이미지, Top10 단어, 연관어 그래프 등).

## 측정

- 환경: 로컬, CPU, Python 3.11, MongoDB Atlas
- 대상: 자막 보유 기사 15건(차트 Top 선정 규모와 유사)
- 스크립트: [`benchmark_caching.py`](../benchmark_caching.py)

| 경로 | 시간 |
| --- | --- |
| **사전계산 + 캐시 조회** (단건 인덱스 조회) | **약 13 ms** |
| 매 요청 분석 — 전처리(Okt, 15건) | 약 7.1 s |
| 매 요청 분석 — KoBERT 추론(15건) | 약 13.2 s |
| 매 요청 분석 — 모델 로드(최초 1회) | 약 2.6 s |
| **매 요청 분석 1회 합계** | **약 20 s** |

## 결론

> 분석을 매 요청 수행 시 **~20초** → 사전계산 후 캐시 조회 시 **~13ms**.
> **약 1,500배 빠른 응답** (보수적으로도 수백 배).

체감상 "기다림이 필요한 분석"을 "즉시 응답"으로 바꾼다.

## 추가 개선 여지(향후)

- `process_text()`가 호출마다 `Okt()`를 새로 생성 → 모듈 레벨 1회 생성으로 전처리 시간 단축 가능
- KoBERT 임베딩 결과 자체를 캐싱하면 재계산 비용 추가 절감
